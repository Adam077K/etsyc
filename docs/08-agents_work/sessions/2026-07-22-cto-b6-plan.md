# B6 dispatch packet — the product page and cart

Plan-only. Nested Task is blocked; the packet is handed to team-lead for dispatch.

## What B6 closes

B6 is the `PRODUCT_PAGE` state in the buyer state machine — the surface a
buyer lands on after tapping a product in a maker's world (B4→B5 hands off
to B6). Today, tapping a product docks the film and swaps narration but
nothing else changes; the interaction is a dead end. B6 renders the
product surface **around** the docked film — title, price, description,
add-to-cart CTA — reserving the §5.3 exclusion zone so the dock never
occludes the CTA on ≥768, and driving the pill-collapse below 768.

The four ACs (AC-B6.DOCK-1..4) are already ruled by CPO in
`docs/08-agents_work/sessions/2026-07-22-cpo-narrate-shrink-ac.md`. This
packet is the engineering decomposition, not a re-litigation of AC.

## Ground truth I established before planning

Confirmed by direct code read (not inferred):

1. **The route slot is free.** `/w/[handle]/p/[productId]` does not exist.
   The world route `/w/[handle]/page.tsx` is the pattern to mirror — it
   uses `getWorld(handle)` cached, `validateStoreConfig` guard, and
   `notFound()` for 404s. `FilmLayerProvider` mounts at app root
   (`app/layout.tsx:58`), so it survives an in-app route transition — the
   film keeps playing across `/w/[handle]` → `/w/[handle]/p/[productId]`
   as long as the layout tree does not remount.

2. **The narration wiring is already there.** `StoreWorld` accepts
   `narrationProductId` explicitly *"to OVERRIDE that wiring (B6: the
   product page renders around the dock and owns which product is open)"*
   (`lib/renderer/StoreWorld.tsx:38-46`). B6 renders `<StoreWorld
   narrationProductId={productId} initialStage="narrate-shrink">`.

3. **The `product-detail` block exists and is 90 % of the composition** —
   `components/blocks/product-detail/index.tsx`. Two-column layout, mono
   price via `formatPrice`, inventory copy, reveal-staggered, add-to-cart
   button. **But:** the CTA has no `data-primary-cta` marker, no click
   handler, no server action wiring. It's rendered as a block inside a
   world; B6 needs a page-scale composition that consumes it (or an
   adapted version of it).

4. **The dock geometry is already declared once, in code**
   (`lib/renderer/dock-geometry.ts`). B6 imports `DOCK_BANDS`,
   `dockBandFor`, `exclusionZoneFor`, `PRIMARY_CTA_MARKER`. It **must
   never re-declare** any of those numbers — this is exactly the class
   the Gate-2 DECISIONS entry (2026-07-22, cookie-attributes
   amendment) locked as compiler-enforced.

5. **The pill-collapse mechanism is NOT shipped.** The B5 session
   (`docs/08-agents_work/sessions/2026-07-22-frontend-engineer-b5.md`)
   says B5 shipped only the corner-rect computation and the **contract**
   (`PRIMARY_CTA_MARKER`, the exclusion-zone table). The pill affordance
   itself does not exist in `FilmLayer.tsx` — no `pill` / `audio-only`
   symbols anywhere in the film module. The CPO ruling wording *"B5
   shipped the mechanism; B6 supplies the trigger element"* is wrong; B5
   shipped the *contract*, and the pill affordance is unbuilt. **B6 must
   build it (or an accompanying B5-fast-follow unit must).** Surfaced as
   OQ #4.

6. **The cart schema is thin on purpose.** `carts` has `id`, `buyer_id`,
   `status`, `created_at`, `updated_at`. **There is no `cart_items`
   table.** `create_order(store_id, items jsonb)` takes items directly on
   the buyer's checkout call — this is by design (`commerce.sql:120`
   header). "Add to cart" between B6 and B7 has three possible
   implementations; see OQ #1.

7. **model3d_id ownership** (adversary P2, `handoff:442`) is an
   Irreversible migration for **S8's write path**, not B6's read path.
   B6 does not write `model3d_id`. **Recommend it does not ride along**
   with B6 — dispatch it as its own unit off the main.

8. **`fix/s8-currency-minor-unit` @ 9f6ef09 is unmerged.** It exports
   `majorToMinor`, `minorToMajor`, `formatPrice`, `currencyExponent`
   from `lib/products/schemas.ts`. B6 only *reads* `products.currency` +
   `price_amount` and displays them via `formatPrice`. On `main`
   @ 2abb982 the S8 branch is merged into `integ/wave3-dryrun` but not
   into `main` — B6 needs the `formatPrice` from S8 (present in
   `product-detail` already, so already assumed available on the base).
   **B6 does not add a currency dependency; it consumes the existing
   `formatPrice` from `lib/utils.ts`/`lib/products/schemas.ts`
   whichever is live on `main`.** Verified `product-detail` currently
   imports from `@/lib/utils` — that path resolves on `main`.

## Migration decision

**No migration required for B6.**

The model3d_id ownership migration (adversary P2, gate-2 handoff line
442) is Irreversible and belongs to S8's write path, not B6's read path.
B6 never writes `model3d_id`. Riding it along would upgrade a Lite
dispatch to Irreversible, forcing Founder sign-off on a change that has
no B6 semantics. **Recommend a separate dispatch cycle for that fix,
with database-engineer + Founder sign-off, off `main`.**

If OQ #1 rules for a DB-backed `cart_items` table, THAT migration is
Irreversible and Founder-signed, and belongs on its own worker
(database-engineer) branching off `main` before U-B starts.

## Open questions — CTO cannot resolve without ruling

### OQ #1 — Cart storage: client-only or DB-backed?

The `carts` table exists as a scratch reference (no items). There is no
`cart_items` table. Three implementations:

- **(a) Client-side (sessionStorage + a lightweight hook)** — no
  migration, ships as Lite, matches `create_order`'s jsonb parameter
  design and the checkout spec's "review items" flow. Cart is lost on
  device change. CTO strongly recommends this for MVP demo.
- **(b) DB-backed `cart_items` table** — Irreversible migration, Founder
  sign-off, cross-device continuity, RLS boundary. Right long-term
  answer; wrong scope for a "close the loop" Wave-3 unit.
- **(c) Hybrid: rows in `carts.metadata jsonb`** — no new table but a
  schema-shape change to `carts` (still Irreversible if a CHECK gets
  added). Worst of both.

**CTO recommendation: (a).** Ruling owner: **CPO** (product decision:
what does "add to cart" promise?). Do not dispatch U-B without this.

### OQ #2 — Post-add-to-cart behavior

After Add to cart succeeds, does the buyer:

- **(a) stay on the product page** with a subtle inline "1 in cart"
  affordance appearing?
- **(b) navigate to `/cart`** (checkout, B7)?
- **(c) see a toast + a "checkout now" pill** and stay on the page?

Design intent (film keeps playing, buyer not yanked) argues for (a) or
(c). B7's checkout spec implies a `/cart` route exists as a destination.

**CTO recommendation: (c) toast + persistent inline "N in cart" pill;
`/cart` route creation belongs to B7.** Ruling owner: **Design-Lead
+ CPO** (visual + product jointly).

### OQ #3 — Reviews block: render empty state or defer entirely?

The product-page spec §Out-of-scope says enriched reviews are B6+
(Wave 4). The `ReviewsBlock` component already exists and takes
`entries?: ReviewEntry[]` defaulting to `[]` — the empty state is a
warm invitation.

**CTO recommendation: render the reviews block with `entries={[]}`.**
Fulfils the "4 states" mandate and the CPO AC line *"an inline compact
trust-badge and the reviews block render inline"*. Ruling owner: **CPO**
(2 lines to confirm or deny).

### OQ #4 — Pill affordance ownership

The CPO ruling states *"B5 shipped the mechanism; B6 supplies the
trigger element."* Direct code inspection contradicts this: B5 shipped
the contract (`PRIMARY_CTA_MARKER`, exclusion zone table, dock rect
geometry) but the pill affordance itself has zero symbols in the film
module. U-C (this packet) adds the affordance to `FilmLayer.tsx`.

**Confirm:** is a B6 unit adding pill code to FilmLayer.tsx acceptable,
or should this be dispatched as a fast-follow B5 fix instead?
Architectural difference: U-C touches a Wave-3-core file with high blast
radius. **CTO recommendation: accept it inside B6's dispatch (U-C) — the
pill affordance has no meaning without B6's trigger and there is no
gain in a separate cycle.** Ruling owner: **CPO** (scope call).

### OQ #5 — Trust badge: inline chips vs `trust-badge` block

`product-detail` currently renders two inline chips (Real maker,
AI-transparency) directly in the block. The product-page spec calls for
the `trust-badge` block in `inline-compact` variant.

**CTO recommendation: keep the inline chips; they satisfy the AC and
avoid duplicate-block-instance ambiguity in a full-page context.**
Ruling owner: **CPO** (2 lines).
